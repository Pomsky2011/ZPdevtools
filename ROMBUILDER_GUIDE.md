# ZeroPoint ROM Builder Guide

The ROM Builder (`rombuilder`) is a tool that combines CPU, PPU, and APU binaries into a single ROM file for the ZeroPoint Fantasy Console.

## Overview

The ROM Builder creates a unified ROM image from three separate binary components:
- **CPU Binary** - DEF88186 main processor code
- **PPU Binary** - Picture Processing Unit microcode
- **APU Binary** - Audio Processing Unit code

It also generates a metadata file (`.txt`) containing:
- CPU entry point address
- Base addresses for each component
- Size information
- Memory layout map

## Installation

```bash
cd ZPdevtools
gcc -o rombuilder rombuilder.c -Wall
```

## Usage

### Basic Usage

```bash
rombuilder -cpu <cpu.bin> -ppu <ppu.bin> -apu <apu.bin> -o <output.rom>
```

### Full Syntax

```
rombuilder [options] -cpu <cpu.bin> -ppu <ppu.bin> -apu <apu.bin> -o <output.bin>

Required:
  -cpu <file>       CPU binary (DEF88186 code)
  -ppu <file>       PPU binary (PPU microcode)
  -apu <file>       APU binary (APU code)
  -o <file>         Output ROM file

Optional:
  -cpu-base <addr>  CPU base address (default: 0x008000)
  -ppu-base <addr>  PPU base address (default: 0x100000)
  -apu-base <addr>  APU base address (default: 0x200000)
  -entry <addr>     CPU entry point (default: cpu-base)
  -v                Verbose output
  -h, --help        Show help message
```

## Default Memory Layout

The ROM Builder uses the following default memory layout:

| Component | Base Address | Max Size  |
|-----------|--------------|-----------|
| CPU       | 0x008000     | 512 KB    |
| PPU       | 0x100000     | 512 KB    |
| APU       | 0x200000     | 512 KB    |

**Total ROM Size:** Up to 16 MB

## Examples

### Example 1: Basic ROM Creation

```bash
# Assemble components
cpuasm main.asm main.bin
ppuasm graphics.asm graphics.bin
apuasm audio.asm audio.bin

# Build ROM
rombuilder -cpu main.bin -ppu graphics.bin -apu audio.bin -o game.rom
```

Output:
```
ZeroPoint ROM Builder
=====================

ROM built successfully!
  Output: game.rom
  Size: 2099200 bytes (2050.00 KB)

ROM is ready to use with the ZeroPoint emulator.
```

This creates:
- `game.rom` - The ROM file
- `game.txt` - Metadata file

### Example 2: Custom Memory Layout

```bash
rombuilder \
  -cpu main.bin \
  -ppu graphics.bin \
  -apu audio.bin \
  -cpu-base 0x010000 \
  -ppu-base 0x080000 \
  -apu-base 0x0C0000 \
  -entry 0x010100 \
  -o custom.rom \
  -v
```

### Example 3: Complete Build Pipeline

```bash
#!/bin/bash
# build_rom.sh - Complete ROM build script

echo "=== ZeroPoint ROM Build Pipeline ==="

# Step 1: Compile C code to assembly
echo "Compiling C code..."
cd src
../../c_compiler/def88186cc main.c lib.c -o ../build/main.asm

# Step 2: Assemble CPU code
echo "Assembling CPU code..."
cd ../build
../../cpuasm main.asm cpu.bin

# Step 3: Assemble PPU code
echo "Assembling PPU code..."
../../ppuasm ../graphics/sprites.asm ppu.bin

# Step 4: Assemble APU code
echo "Assembling APU code..."
../../apuasm ../audio/music.asm apu.bin

# Step 5: Build ROM
echo "Building ROM..."
../../rombuilder \
  -cpu cpu.bin \
  -ppu ppu.bin \
  -apu apu.bin \
  -entry 0x8000 \
  -o game.rom \
  -v

echo ""
echo "=== Build Complete ==="
echo "ROM: build/game.rom"
echo "Metadata: build/game.txt"
```

## Metadata File Format

The ROM Builder automatically generates a `.txt` metadata file with the same base name as the ROM:

```ini
# ZeroPoint ROM Metadata
# Generated by rombuilder

[ROM]
file = game.rom

[CPU]
binary = main.bin
base = 0x008000
size = 32775
entry = 0x008000

[PPU]
binary = graphics.bin
base = 0x100000
size = 4096

[APU]
binary = audio.bin
base = 0x200000
size = 2048

[Memory Layout]
# Component   Base        End         Size
# CPU         0x008000    0x010007    32775 bytes
# PPU         0x100000    0x101000    4096 bytes
# APU         0x200000    0x200800    2048 bytes
```

This metadata file is used by:
- The ZeroPoint emulator to load the ROM correctly
- Debugging tools to understand memory layout
- Documentation and analysis

## Integration with C Compiler

### Option 1: Manual Pipeline

```bash
# Compile C to assembly
def88186cc main.c -o main.asm

# Assemble to binary
cpuasm main.asm cpu.bin

# Build ROM (with existing PPU/APU binaries)
rombuilder -cpu cpu.bin -ppu graphics.bin -apu audio.bin -o game.rom
```

### Option 2: Using -bin Flag

```bash
# Compile and assemble C code in one step
def88186cc main.c -bin -o cpu.bin

# Build ROM
rombuilder -cpu cpu.bin -ppu graphics.bin -apu audio.bin -o game.rom
```

## Memory Layout Customization

You can customize where each component is loaded in memory:

### Example: Compact Layout (Low Memory)

```bash
rombuilder \
  -cpu main.bin -cpu-base 0x004000 \
  -ppu gfx.bin  -ppu-base 0x008000 \
  -apu snd.bin  -apu-base 0x010000 \
  -o compact.rom
```

### Example: Banking System

```bash
# CPU code in first bank
# PPU in second bank
# APU in third bank
rombuilder \
  -cpu main.bin -cpu-base 0x000000 \
  -ppu gfx.bin  -ppu-base 0x010000 \
  -apu snd.bin  -apu-base 0x020000 \
  -o banked.rom
```

## Entry Point

The CPU entry point specifies where the DEF88186 processor begins execution.

**Default:** Same as CPU base address

**Custom Entry Point:**
```bash
# CPU code starts at 0x8000, but execution begins at 0x8100
rombuilder \
  -cpu main.bin \
  -cpu-base 0x8000 \
  -entry 0x8100 \
  -ppu gfx.bin \
  -apu snd.bin \
  -o game.rom
```

This is useful when:
- You have initialization code before the main entry point
- Your assembler places the entry point at a specific location
- You want to skip a header or data section

## Verification

After building a ROM, verify it:

```bash
# Check file size
ls -lh game.rom

# View metadata
cat game.txt

# Verify with hexdump (check CPU code at base address)
hexdump -C game.rom | grep "00008000" -A 5
```

## Troubleshooting

### Error: "Cannot open input file"

**Problem:** Source binary file doesn't exist

**Solution:** Check that you've assembled all components first:
```bash
ls -l main.bin graphics.bin audio.bin
```

### Error: "File is too large"

**Problem:** One component exceeds 512 KB

**Solution:**
- Optimize your code to reduce size
- Split into multiple files using `.include`
- Adjust base addresses to allow more space

### Error: "Would exceed ROM size"

**Problem:** Component placement would exceed 16 MB ROM limit

**Solution:** Adjust base addresses to use lower memory regions

### Warning: "Cannot create metadata file"

**Problem:** Write permissions issue

**Solution:** Check directory permissions:
```bash
ls -ld .
chmod u+w .
```

## Best Practices

1. **Use Consistent Naming**
   ```
   game_cpu.bin
   game_ppu.bin
   game_apu.bin
   game.rom
   ```

2. **Keep Metadata Files**
   - Commit `.txt` files to version control
   - They document your ROM's memory layout
   - Useful for debugging and team collaboration

3. **Use Build Scripts**
   - Automate the entire pipeline
   - Reduces errors
   - Ensures reproducible builds

4. **Version Your ROMs**
   ```bash
   rombuilder ... -o game_v1.0.rom
   rombuilder ... -o game_v1.1.rom
   ```

5. **Test After Building**
   - Load ROM in emulator immediately
   - Check that all components work
   - Verify entry point is correct

## Integration with Make

```makefile
# Makefile for ZeroPoint ROM

CPU_SRC = src/main.c src/lib.c
PPU_SRC = graphics/sprites.asm
APU_SRC = audio/music.asm

ROM = game.rom

all: $(ROM)

cpu.bin: $(CPU_SRC)
	cd c_compiler && ./def88186cc $(CPU_SRC) -bin -o ../cpu.bin

ppu.bin: $(PPU_SRC)
	./ppuasm $(PPU_SRC) -o ppu.bin

apu.bin: $(APU_SRC)
	./apuasm $(APU_SRC) -o apu.bin

$(ROM): cpu.bin ppu.bin apu.bin
	./rombuilder -cpu cpu.bin -ppu ppu.bin -apu apu.bin -o $(ROM) -v

clean:
	rm -f cpu.bin ppu.bin apu.bin $(ROM) $(ROM:.rom=.txt)

.PHONY: all clean
```
